> 设计人：高见远（架构师）| 日期：2026-06-25
> 基于：优化 PRD v1.0（产品经理 许清楚）

---

## 一、架构设计目标

基于 PRD 的三轮迭代规划，架构设计需要满足：

1. **安全性** - 解决密码明文、匿名登录、无权限控制等 P0 安全问题
2. **可维护性** - 从单文件拆分为模块化结构，支持团队协作
3. **数据可靠性** - 统一数据层，消除 localStorage/CloudBase 双轨混用
4. **可扩展性** - 为后续功能增强（评论、标签、通知等）预留扩展点

### 2.3 认证方案

**方案：CloudBase 自定义登录 + 云函数验证**

```
用户输入账号密码
       │
       ▼
┌──────────────┐     HTTPS      ┌─────────────────┐
│   前端登录    │ ──────────────→│  云函数: login   │
│   表单提交    │                │                  │
└──────────────┘                │  1. 查询 users   │
                                │  2. 验证密码哈希 │
                                │  3. 生成 token   │
                                └────────┬────────┘
                                         │
                                         ▼
                                ┌─────────────────┐
                                │  返回 token      │
                                │  + 用户信息      │
                                └─────────────────┘
                                         │
                                         ▼
                                ┌─────────────────┐
                                │  前端存储 token  │
                                │  设置过期时间    │
                                └─────────────────┘
```

## 四、任务分解

### 第 1 轮迭代：安全与基础（P0）

#### 任务 1.1：密码安全化 [QA-05]
- **复杂度**: 中
- **前置条件**: 无
- **实现内容**:
  1. 创建 `users` 集合
  2. 编写云函数 `login/index.js`，实现密码哈希验证
  3. 修改前端 `auth.js`，调用云函数登录
  4. 删除前端硬编码的 USERS 对象
- **验收标准**: 密码不再出现在前端代码中

#### 任务 1.2：CloudBase 权限规则 [QA-06]
- **复杂度**: 低
- **前置条件**: 任务 1.1
- **实现内容**:
  1. 配置 demands 集合安全规则：仅认证用户可读写
  2. 配置 reviews 集合安全规则
  3. 配置 schedules 集合安全规则
- **验收标准**: 匿名用户无法访问数据

#### 任务 1.3：排期数据云端同步 [FE-06, QA-10]
- **复杂度**: 中
- **前置条件**: 任务 1.2
- **实现内容**:
  1. 创建 `schedules` 集合
  2. 实现 `services/schedule-service.js`，使用 CloudBase
  3. 编写数据迁移脚本（localStorage → CloudBase）
  4. 修改排期页面使用新 service
- **验收标准**: 排期数据在多设备间同步

#### 任务 1.4：原子化数据保存 [QA-09]
- **复杂度**: 中
- **前置条件**: 任务 1.2
- **实现内容**:
  1. 重构 `db.js`，支持单条 upsert 操作
  2. 修改 `demand-service.js` 的保存逻辑
  3. 添加乐观锁 version 字段
- **验收标准**: 保存操作不再"全删全写"

#### 任务 1.5：需求状态流转管控 [FE-01, BP-01, BP-02]
- **复杂度**: 中
- **前置条件**: 任务 1.4
- **实现内容**:
  1. 定义状态流转规则配置
  2. 实现 `models/demand.js` 的状态验证
  3. 修改编辑弹窗，根据当前状态限制可选状态
  4. 添加状态变更必填字段校验
  5. 记录 statusHistory
- **验收标准**: 非法状态跳转被阻止

#### 任务 1.6：卡点超时提醒 [BP-07, FE-09]
- **复杂度**: 低
- **前置条件**: 任务 1.3
- **实现内容**:
  1. 在排期列表渲染时计算卡点天数
  2. 超 3 天标红，超 7 天标深红
  3. 首页加载时弹窗提醒超 7 天卡点
- **验收标准**: 卡点超时有明显视觉提示

#### 任务 1.7：需求列表分页 [UX-08]
- **复杂度**: 低
- **前置条件**: 无
- **实现内容**:
  1. 实现 `components/table.js` 分页功能
  2. 修改 `list-page.js` 使用分页组件
  3. 默认每页 20 条，支持切换
- **验收标准**: 100 条数据首页加载 < 1 秒

#### 任务 1.8：表单字段分组折叠 [UX-04]
- **复杂度**: 低
- **前置条件**: 无
- **实现内容**:
  1. 将表单字段分为"基础信息"和"扩展信息"
  2. 扩展信息默认折叠
  3. 添加展开/折叠切换按钮
- **验收标准**: 首屏仅展示 6-8 个核心字段

#### 任务 1.9：网络异常处理 [QA-13]
- **复杂度**: 低
- **前置条件**: 无
- **实现内容**:
  1. 封装 `db.js` 的错误处理
  2. 添加重试机制（最多 3 次）
  3. 超时自动降级为 localStorage
  4. 友好错误提示
- **验收标准**: 网络异常时用户看到明确提示

#### 任务 1.10：代码模块化拆分 [QA-01, QA-02, QA-03]
- **复杂度**: 高
- **前置条件**: 无（可与其他任务并行）
- **实现内容**:
  1. 拆分 CSS 到 4 个文件
  2. 提取 `utils/constants.js`（国家、模块、PM 列表）
  3. 提取 `utils/date.js`、`utils/html.js`
  4. 实现 `core/db.js` 统一数据层
  5. 实现 `core/auth.js`
  6. 实现 `core/router.js`
  7. 提取各页面到 `pages/` 目录
  8. 提取组件到 `components/` 目录
  9. 清理 config.js 中的 Supabase 配置
- **验收标准**: 单文件 < 200 行，模块间低耦合
